## [Unreleased]

### Fixed
- `src/model/metrics.py:compute_ssim`: validation 끝에서 SSIM이 GPU OOM 나던 문제 (172 GB 점유 + SSIM intermediate 5+ GB 필요 → 2.79 GB free에 안 들어감). N×3×H×W 전체에 대해 한 번에 호출하던 것을 chunk=8로 분할 loop으로 변경. chunk별 intermediate ~200-400 MB로 줄여서 free 메모리에 fit. 결과는 sample 수 가중 평균 (이전 single-call의 batch 평균과 수학적 동일).
- `src/model/denoiser/wan_ti2v.py` `channel_concat` patch_embedding extra channel 수: `pose_embed(skip_embedding=True)`가 `embedding.in_channels * temporal_downsample = 6 * 4 = 24` 채널을 반환하는데 `__init__`에서 patch_embedding을 `+6`만 확장해서 conv가 `expected 54, got 72` 에러. `extra_in = 6 * max(temporal_downsample, 1)`로 수정 (wan_control의 SimpleAdapter `in_dim`과 동일 규칙).
- `src/model/denoiser/wan_ti2v.py:simple_wan_video_fn` scene cross_attention concat: text context와 scene_context를 dim=1으로 cat할 때 batch dim mismatch 발생하던 버그. CFG의 uncond branch가 `""`을 batch=1로 인코딩하는 반면 scene_context는 val/train batch (예: 8)를 갖고 있어 `RuntimeError: Sizes of tensors must match except in dimension 1`. cat 직전에 broadcast/repeat_interleave 추가 (한쪽이 1이면 expand, 한쪽이 다른쪽의 배수면 repeat).

- `src/model/denoiser/wan_ti2v.py`: `camera_input_type="ac3d"` 모드 추가 — AC3D / CogVideoX-ControlNet 패턴으로 카메라 주입. `__init__`에서: (a) `model.ac3d_patch_embedding` Conv3d (in_ch = orig + 6*T = 72), 첫 48ch는 pretrained patch_embedding 복사 + 새 24ch zero-init; (b) `model.ac3d_blocks` = 첫 `ac3d_num_layers` (기본=2) 개 main Wan block을 `NewDiTBlock.from_dit_block(..., camera_input_type="none", scene_input_type="none")`로 deepcopy (내부 scene/recam path 비활성); (c) `model.ac3d_projectors` = N개 zero-init `nn.Linear(dim, dim)`. `simple_wan_video_fn`은 main DiT vanilla patchify는 그대로 두고, 별도로 `ac3d_patch_embedding(cat(latent, ray))` → `ac3d_blocks` → `ac3d_projectors`로 layer별 residual 산출, main block loop에서 `i < ac3d_num_layers`이면 `x[:, :n_patches] += ac3d_residuals[i]`로 주입 (scene_latent_tokens 부분은 보존). zero-init projector 덕분에 t=0 시점 vanilla Wan과 동일. `_set_trainable_parameters`는 `ac3d_*` 전부 trainable. `WanTI2VConfig.ac3d_num_layers: int = 2` 필드 추가. shell 사용: `+model.denoiser.camera_input_type=ac3d +model.denoiser.ac3d_num_layers=2 +model.denoiser.camera.input_shape=[H_latent,W_latent]`. AC3D 원본의 timestep gating (t ∈ [800, 1000]) 는 아직 미구현 — wrapper의 noise sampler 변경 필요시 별도.

### Changed
- `train_ti2vgen_recon_overfit_256_finetuned.sh`: view_sampler를 `dataset/view_sampler=caption_window`로 전환 + `+dataset.load_prompts=true` 추가. caption 구간에 정렬된 연속 target window로 학습하고 해당 caption이 `sample["text"]`로 들어가게 함. `model.text_encoder=null`이지만 wan_ti2v denoiser가 내부 UMT5(`uses_internal_text_encoder=True`)로 `batch["text"]`를 인코딩해 cross-attention context로 주입하므로 text 컨디셔닝은 정상 동작 (`scripts/verify_caption_window_text.py`로 data 경로 검증 완료).
- `.gitignore`에 vendored upstream repo 3개 추가: `src/model/DiffSynth-Studio` (~417GB), `src/model/ac3d`, `src/model/GEN3C`. 각각 nested `.git`을 가진 read-only 참조라 `git add -A` 시 인덱스에 끌려가는 사고 방지 (`WorldTraj`는 이미 등록돼 있었음).
- `CLAUDE.md`에 **Git 커밋 워크플로우** 섹션 추가. 역할 분담 명문화: Claude는 코드 변경 작업마다 분기 없이 **현재 브랜치에** 변경+CHANGELOG를 하나의 커밋(`feat/fix/exp/chore: <요약>` + Co-Authored-By)으로 묶고 요약 보고 후 멈춤 (요청 없이 자동 커밋 — 기본 "요청 시에만 커밋" + "default 브랜치면 분기" 동작 override); push/merge/pull/브랜치 생성·삭제는 사용자 수동. 스테이징은 변경 파일만 명시적 `git add <경로>` (`git add -A`/`.` 금지) — vendored 디렉토리(`src/model/{DiffSynth-Studio,ac3d,GEN3C}` ~417GB, `WorldTraj`)가 인덱스에 끌려가는 사고 차단. 읽기전용/분석 작업은 커밋 미수행.
- `CLAUDE.md` 업데이트 — `/init` 결과. 코드 드리프트 반영: `camera_input_type`에 `channel_concat`/`ac3d`, `condition_latents_input_type`에 `first_frame_depth`/`first_frame_depth_soft`, 새 `scene_projection` ∈ `{linear, mlp}` 노브 추가. Models 섹션에 vendored `DiffSynth-Studio`/`ac3d`/`GEN3C` 디렉토리 (read-only 참조) 명시. Data 섹션에 dl3dv/re10k **scene blacklist** 메커니즘 (`blacklist.csv`, helper 위치, DL3DV NaN 자동 append vs re10k 수동) 추가. Commands에 `fast_infer_t2v.sh` standalone inference, env vars에 `src/main.py`의 `OMP_NUM_THREADS` setdefault 추가.
- `src/main.py`: 모든 import 이전에 `os.environ.setdefault("OMP_NUM_THREADS", "8")` + MKL/OpenBLAS 동일. PyTorch default가 모든 코어를 intra-op parallel pool로 잡아서 main process 1개가 72-core 머신에서 4000%+ CPU spike을 내고, train job 4개 동시 실행 시 core 경쟁으로 throughput이 떨어지던 문제. `setdefault`라서 shell에서 명시적으로 export하면 override 가능. 효과는 fork되는 dataloader worker에도 자동 전파. 적용 범위는 `src.main` 통해 들어오는 모든 train/infer (SceneGen `src.main_scene`은 별도 적용 필요).

### Added
- `scripts/probe_lpips_fp16.py` — LPIPS metric이 학습 중 같은 값으로 수렴하는 증상 진단용 probe. (1) noise scale 0.001–0.05에서 fp32/fp16 LPIPS 비교 — fp16이 정상으로 monotonic하게 변하고 fp32와 1e-4 이내 일치하는 것 확인. (2) lin weight nonzero ratio 출력 — fp16 캐스트 후에도 underflow 없음(`14716138/14716160 ≈ 1.0`). (3) `tmp/wan_vae_recon/`의 real Wan VAE recon 10쌍에 대해 `Metric()`을 wrapper-동일 setup(`freeze` + `convert_to_buffer`)으로 instance화해서 bf16 입력으로 `compute_lpips` 호출 — per-scene `0.0123~0.0447`로 정상 변화, aggregate `0.024292`가 fp32 평균(`0.024125`)과 0.0002 이내 일치. 결론: **compute_lpips 함수 자체는 정상**. 남은 가설은 LightningModule의 precision-cast가 LPIPS buffer를 다운캐스트하는지 (H6).
- `scripts/dump_va_vae_recon.py` — VA-VAE (`checkpoints/vavae-imagenet256-f16d32-dinov2.pt`, f=16, latent_channels=32) 버전의 recon dump. 동일한 10개 re10k test scene을 **세 해상도**(256×256, 256×448, 480×832)에서 encode→decode 후 `tmp/va_vae_recon/`에 저장. 입력은 `(B,C,H,W)`, `(x*2-1)`로 [-1,1] 정규화 후 `.latent_dist.mode()`로 deterministic latent 사용. 결과: 256×256은 PSNR **31.40 dB** / L1 0.0176, 256×448은 33.67 dB / 0.0141, 480×832는 37.70 dB / 0.0088 — PSNR이 latent 토큰 수(256→448→1560)와 monotonically 비례. **VA-VAE의 학습 native 256×256에서도 recon이 가장 나쁨** (학습 해상도 근접성보다 f=16 bottleneck의 spatial 토큰 수가 dominant). Wan VAE도 같은 방향 (256×448→480×832에서 PSNR 6.4 dB 갭). VA-VAE는 raw latent magnitude가 큼 (z_std≈3.5, |z|max 최대 30) → 학습 경로의 `latent_mean`/`latent_std` 정규화 필요. |z|max는 큰 해상도에서 오히려 더 큼 → 작은 해상도 outlier 가설은 양 VAE에서 모두 지지되지 않음.
